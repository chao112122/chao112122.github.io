edr规避技术-NtCreateUserProcess,PPID欺骗与BlockDLLs技术对抗

1.前言

在现代红队评估中，传统的 CreateProcess已经成为了 EDR 监控的重灾区。现代化c2的命令执行都是使用CreateProcess调用cmd.exe携带参数执行,一旦执行次数过多,在edr视角就是一个进程不断创建cmd进程,是十分敏感的操作,后续衍生了bof技术,但今天提及的是NtCreateUserProcess,他是CreateProcess的底层调用,其次对于父子进程,也是edr的重点观察对象。

为了实现更深层级的隐匿，我们需要跳出 Win32 子系统的限制，直接与 Native API 对话。本文将分享如何通过 ntdll!NtCreateUserProcess 同时实现 PPID Spoofing（父进程欺骗） 与 BlockDLLs（非微软签名限制策略），从底层切断 EDR在r3 的监控链条。

2.NtCreateUserProcess与BlockDLLs原理实现

NtCreateUserProcess原理:

大部分 EDR 会在 kernel32.dll 或 kernelbase.dll 的高层函数上挂钩。而 NtCreateUserProcess 是用户态进入内核前的最后一站，它的参数不再是易读的 STARTUPINFO 结构，而是基于 PS_ATTRIBUTE_LIST 的属性列表。

这种结构的优势在于：

  1. 隐蔽性：避开了高层 API 的常规监控。
  2. 灵活性：可以通过增删属性块，在一次系统调用中完成多种对抗配置。

核心机制：PS_ATTRIBUTE_LIST

这是该技术的灵魂。PS_ATTRIBUTE_LIST 本质上是一个头部结构加上一个 PS_ATTRIBUTE 数组。

在手动构造时，必须严格处理 TotalLength。如果计算偏移不当，内核会直接返回 STATUS_INVALID_PARAMETER 或 122 (ERROR_INSUFFICIENT_BUFFER)。

BlockDLLs原理：

EDR 极度依赖用户态 Hook，而 Hook 通常是通过注入非微软签名的 DLL 实现的。 开启 PROCESS_CREATION_MITIGATION_POLICY_BLOCK_NON_MICROSOFT_BINARIES_ALWAYS_ON 策略后，系统会拒绝加载任何非微软签名的二进制文件。

实战意义： 即使进程被发现，EDR 的 DLL 也无法进入你的进程空间，从而失去了对你 API 调用的感知能力。

3.ppid技术

许多安全工具（如入侵检测系统、日志分析系统、反病毒软件等）会基于进程的父子关系进行异常检测。如果一个恶意进程的父进程与其行为不符，通常会触发报警。比如，一个恶意进程如果不是由explorer.exe等核心进程创建，而是由另一个恶意进程创建，可能会被安全系统识别为异常，导致攻击者暴露。

通过PPID欺骗，攻击者可以修改进程的父进程ID，使其看起来像是由一个无害的父进程（例如操作系统的关键进程或用户信任的程序）创建，从而绕过安全系统的监控。这种欺骗行为让恶意进程能够在不引起警觉的情况下运行，从而延长其生存时间。
