前言

在 Windows 架构中，杀软内核驱动通常扮演着“超级裁判”的角色。它通过在内核中植入一系列通知回调（Notification Routines）和微过滤监听器（Minifilter Callbacks），将自己强行插入到系统处理关键动作（如进程启动、文件读写、内存映射）的路径上。

分析这些驱动的概念核心可以归纳为以下三个维度：

监控点的“锚定”与“劫持”

杀软并不生成系统动作，它只“订阅”动作。分析的首要目标是确认它在内核中注册了哪些眼线。

注册回调：利用 PsSetCreateProcessNotifyRoutine 等标准接口，确保任何新进程的诞生都必须先经过它的逻辑判断。

对象过滤：通过 ObRegisterCallbacks 监控句柄操作，防止恶意程序尝试打开 LSASS 等敏感进程的句柄。

文件过滤：利用 Minifilter 框架，在文件到达磁盘前进行扫描，这是拦截勒索软件或恶意载荷（Payload）落地的核心战场。

策略池的“匹配”与“决策”

当一个动作被拦截下来后，驱动内部会进入逻辑决策阶段。

静态特征匹配：将当前进程名、路径或证书信息与内存中的全局规则链表（如你之前看到的 MpBmDocOpenRules）进行快速比对。

动态行为评分：根据进程短时间内的 API 调用序列（如：申请内存 -> 修改权限 -> 远程注入）进行启发式评估。

内核态与用户态的通信：内核驱动（EDR Driver）通过 FilterCommunicationPort 将复杂的判定任务“外包”给用户态的高性能分析服务，并根据返回的结果决定是否给该动作判“死刑”。

“阻断”与“放行”的终极对抗

分析的终点在于理解杀软如何下达指令。

状态码注入：在回调中将 CreationStatus 修改为 STATUS_ACCESS_DENIED，从内核底层撤销该动作的合法性。

上下文擦除：在恶意代码执行前，通过抹除其内存映射或强制结束其线程池来实现无感阻断。

创建进程回调

在内核中，杀软通过 PsSetCreateProcessNotifyRoutineEx 注册回调。每当一个新的 EPROCESS 被分配并准备运行时，内核会暂停进程，并挨个询问杀软进程的安全性,杀软会对多个角度进行检查,在后续我们也提到了

我们在windbg查看进程创建回调

1.文件名检查

可以看到在

fffff80a`83ab8d6c mov rbx, qword ptr [rdi+30h] ; 获取 ImageFileName 指针

fffff80a`83ab8d8f call WdFilter!MpGetImageNormalizedName

Defender 并不直接信任 rdi+30h 里的路径，而是调用 MpGetImageNormalizedName 进行格式化。这是为了防止通过路径欺骗（如符号链接、长路径名等）绕过检测。这里可以大概判断window defender可能对文件名是有鉴定的 可能是为了防止一些红队钓鱼手法?

2.恶意文件处理

在fffff80a`83ab8ea2 c74740220000c0 mov dword ptr [rdi+40h],0C0000022h

这里rdi就是_PS_CREATE_NOTIFY_INFO结构 偏移40h就是CreationStatus 而0xC0000022刚才就是拒绝访问的状态码 也就是说是这个判断进程是否安全

后面代码太长就不放上来了 好多一直call和jmp不停 看得我头大 就只是简单看一下

3.父进程上下文检测

fffff80a`83ab921e call WdFilter!MpGetProcessContextById

fffff80a`83ab9230 bts dword ptr [rcx+34h], 0Fh

这里前面获得了父进程id 然后赋值给r13 call了MpGetProcessContextById 应该是在检测父进程的上下文

4.文档检测

fffff80a`83ab9241 call WdFilter!MpSetProcessDocOpenRule

这里有点意外 虽然知道杀软都会防止宏病毒之类的 这里他会回溯父进程 看是不是文档编辑器 或者检测子进程是不是唤起cmd powershell啥的

可以看到 msaccess.exe powerpoint.exe acrord32.exe都是被检测的对象

5.传回用户层

fffff80a`83ab93e4 call WdFilter!MpSendProcessMessage

这里没搞明白 可能是传回用户层做云端查杀和启发式

6.特殊进程

fffff80a`83ab92ab call WdFilter!MpSetProcessExempt

fffff80a`83ab92ba call WdFilter!MpSetProcessHardening

MpSetProcessExempt是判断是否为关键进程 例如系统进程

MpSetProcessHardening是相关策略 例如禁止加载非微软签名dll

7.检测创建进程的调用者

fffff80a`83ab9300 call qword ptr [WdFilter!_imp_IoThreadToProcess]

fffff80a`83ab930c cmp rax, rbx

这个应该是防止远程创建

8.白名单逻辑

fffff80a`83ab6268 call qword ptr [WdFilter!_imp_FltParseFileName] 拿到全路径

fffff80a`83ab62a9 call qword ptr [WdFilter!_imp_RtlEqualUnicodeString] 对比是否为白文件

fffff80a`83ab62c1 call qword ptr [WdFilter!_imp_RtlPrefixUnicodeString] 对比是否在白目录下

9.设置

1: kd> dq [fffff80a`83a77990]+18 L1

fffff80a83a779a8 000013880000ea60

这里000013880000ea60就是defender默认扫描超时时间 60秒

线程回调检测

前言：被锁死的 Ring3 注入

在现代红队攻防中，传统的 CreateRemoteThread 或 QueueUserAPC 注入早已不再是单纯的 API 调用。Windows 内核提供的 PsSetCreateThreadNotifyRoutine 回调成了所有杀软的强力检查手段 ,很多时候线程创建基本不可能绕过了,基本只能靠发现新的滥用手法,例如前年的pool注入 我们在winbg查看注册的线程回调

PspSetCreateThreadNotifyRoutine

PsSetLoadImageNotifyRoutineEx

PsSetCreateProcessNotifyRoutineEx2

分别先后调了这些

通过代码可以定位到

线程回调数组 (Thread):fffff800’171dc5a6 488d0dc3d3daff lea rcx,[nt!PspCreateThreadNotifyRoutine (fffff800’16f89970)]

地址：fffff800’16f89970

可以看到有5个驱动在监控线程创建

分别是

重点关注WdFilter!MpCreateThreadNotifyRoutine和WdFilter!MpCreateThreadNotifyRoutineEx

(1) 逻辑入口 A：fffff80a’83aaf690 (MpCreateThreadNotifyRoutine)

这个函数通常负责基础过滤。

主要工作： 检查 MpData,如果创建者是白名单直接返回。

审计重点： 寻找 cmp 指令比对 MpData+0xF0 的地方。

(2) 逻辑入口 B：fffff80a’83ab5db0 (NotifyRoutineEx)

这个函数是关键,它能拿到 PS_CREATE_THREAD_NOTIFY_ROUTINE_EX 结构，里面包含线程的初始栈空间和起始地址。

主要工作： 进行 VAD扫描。

检测点： 它会检查 StartAddress 是否指向一个 SEC_IMAGE 映射。如果你是用 VirtualAlloc 申请内存后直接 CreateRemoteThread，这个函数会立刻识破你

fffff80a`83aaf6d8 4183fc04 cmp r12d,4这一句以及后面的代码证明了system进程创建线程是不管的 因为4刚好就是system进程

这里也是检查是远程还是本地创建线程

镜像加载回调数组 (LoadImage):fffff800’171dc651 488d0d18d5daff lea rcx,[nt!PspLoadImageNotifyRoutine (fffff800’16f89b70)]

地址：fffff800’16f89b70
