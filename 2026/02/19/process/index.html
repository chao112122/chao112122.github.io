权限不对等：service.exe 以 SYSTEM 权限运行，它拥有修改系统关键文件（如 DLL、驱动、EXE）的最高权力。

目标位置可控：该服务试图写入一个普通用户也能读写的目录（如 C:\Temp\update.log）。

符号链接（Symlink）劫持：

  • 你无法直接修改 C:\Windows\System32\uxtheme.dll。
  • 但你拥有 C:\Temp\update.log 的控制权。
  • 你利用工具（如 Google Project Zero 的 CreateSymlink.exe）创建一个符号链接，让 C:\Temp\update.log 在操作系统底层指向 C:\Windows\System32\uxtheme.dll。

触发写入：当服务再次执行“写日志”操作时，Windows 的 I/O 管理器会沿着你的链接走。服务以为自己在写日志，实际上它把日志内容（或者它控制的任何数据）覆盖到了系统 DLL 上。

为什么“写日志”能导致 LPE？

你可能会问：“日志内容通常是不可控的字符串，覆盖了 DLL 难道不是直接导致程序崩溃吗？”

确实，单纯的日志内容覆盖通常会导致 DoS (拒绝服务)。但从“写文件”到“提权”，通常有以下几种进阶路径：

1. 任意内容写入 (Arbitrary Content Write)

如果日志内容中有一部分是用户可控的（例如：服务记录了尝试登录的用户名），你可以把恶意代码（Shellcode）作为用户名输入。当服务写日志时，它就把你的恶意代码写进了系统 DLL。

2. 劫持执行流 (DLL Hijacking / Replacement)

这是最常见的手法。通过覆盖一个系统启动时必载入、但非核心校验的 DLL（如 uxtheme.dll 或某些第三方服务的 DLL）：

  • 覆盖：将原本合法的 DLL 替换为你编译的恶意 DLL（如果你能控制写入的内容）。
  • 重启/触发：系统重启或相关服务重启时，会加载这个被篡改的 DLL。由于加载者是 SYSTEM 进程，你的恶意代码就在 SYSTEM 权限下执行了。

3. 配置文件/注册表劫持

如果不是写日志，而是写入配置文件（如 .ini 或 .xml），你可以修改其中的路径参数，让系统去执行你放在 C:\Users\Public 下的木马。
