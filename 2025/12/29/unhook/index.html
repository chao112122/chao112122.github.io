edr规避技术-ntdll的unhook技术

前言

在现代红队评估中，EDR（Endpoint Detection and Response）几乎是避不开的槛。大多数 EDR 都会在r3层hook常见的dll,最重要的莫过于ntdll,因为大多数函数调用最后都必须经过ntdll，在 ntdll.dll 的关键系统调用（Syscall）入口处强行插入 jmp 指令，从而监控我们的 API 调用。

与其研究怎么绕过每一个具体的 Hook，不如推倒重来。今天聊聊如何通过动态映射磁盘原件，实现对内存 ntdll 节区的全量覆盖，从底层彻底清除 EDR 的监控点,但这种技术只能只能除去r3的,对于edr的内核回调检测还是无法解决.

本文将体系化地讨论两种主流的脱钩ntdll的技术：

  1. 方案 A：磁盘映射—— 从文件系统获取原件。
  2. 方案 B：傀儡进程克隆 —— 从挂起进程内存获取原件。

方案 A

核心思路：从“干净”的源头借力

我们的目标是将内存中被污染的 .text 段替换为磁盘上原始的 .text 段。为了保持操作的优雅和高效，我们利用 Windows 的 SEC_IMAGE 特性：

  1. 获取基址：利用 GetModuleInformation 定位当前进程中 ntdll 的加载地址。
  2. 内存映射：以 SEC_IMAGE 标志映射磁盘上的 ntdll.dll。这样系统会自动按照 PE 节区的虚拟地址（RVA）帮我们对齐，省去了手动计算文件偏移的麻烦。
  3. PE 寻址：通过解析 DOS 头和 NT 头，精准定位 .text 节的位置和大小。
  4. 权限修补与覆盖：利用 VirtualProtect 临时开启 RWX 权限，直接 memcpy 覆盖。

方案 B

技术原理:窃取合法进程的干净ntdll

核心思想很简单：创建一个处于挂起状态的合法系统进程（如 notepad.exe）。此时，该进程的 ntdll.dll 已经由内核加载器（Ldr）映射进内存，但由于进程尚未正式运行，EDR 的应用层 Hook 往往还未来得及植入。

我们直接通过 ReadProcessMemory 将这个“纯净版”的 ntdll 偷过来，覆盖掉自己进程里被污染的 .text 段。

深度总结：Unhooking 的实战意义与技术博弈

通过对“磁盘映射（SEC_IMAGE）”与“傀儡进程克隆（Perun’s Fart）”这两种技术的复盘，我们可以清晰地看到红队对抗演进的脉络。

1. 为什么我们必须使用 Unhooking 技术？

在现代 EDR 面前，直接调用 NtCreateThreadEx 或 NtWriteVirtualMemory 无异于自投罗网。EDR 通过在 ntdll 的 Syscall 入口处强行插入 jmp 指令，建立了一个应用层的“检查站”。 Unhooking 的核心意义在于：重新夺回对底层指令的控制权。 只有恢复了原始的 syscall 存根，我们的恶意载荷才能真正绕过那些基于用户态 API 监控的特征策略，让调用直接通向内核。

2. 方案优势深度对比

在体系化的实战评估中，没有“最好”的技术，只有“最合适”的策略：

  • 磁盘映射 (Disk Mapping)：
    • 优势：实现极其简单。利用 SEC_IMAGE 特性，内核会帮我们处理好所有的节区对齐和内存映射。
    • 局限：对 C:\Windows\System32\ntdll.dll 的读取操作过于“显眼”。对于部署了严密 Minifilter 驱动的 EDR，这种行为比较明显
  • 傀儡进程克隆 (Suspended Process)：
    • 优势：实现了真正的“无文件”脱钩。通过读取合法挂起进程（如 notepad.exe）的内存，我们避开了磁盘 IO 审计，且利用了 ASLR 在系统级 DLL 上的全局一致性，隐蔽性极强。
    • 局限：CreateProcess 配合 CREATE_SUSPENDED 本身是一个高频触发行为审计（EDR Telemetry）的动作，需要配合 Parent PID Spoofing 等技术来增加欺骗性。

3.讨论

实际这两个unhook技术在现代化的edr还是比较弱的 很多edr已经能够抓这两种技术的路径去检测 对于现代化的edr 使用更加冷门的手法或者syscall的技术更加合适
